
## Preparing for Node Maintenance

### Node Maintenance

To prepare a node for maintenance, cluster administrators must first __*Cordon*__ the node to prevent the cluster from deploying new workloads, then __*Drain*__ it to move the current workload.

To cordon the node

```bash
oc adm cordon <NodeName>
```

To check the node is cordoned

```bash
oc get node <NodeName>
```
```bash

node2      <b>Ready,SchedulingDisabled</b>   control-plane,master,worker    4d20h   v1.31.6
```

To drain the node
```bash
oc adm drain <NodeName> \
  --delete-emptydir-data \
  --ignore-daemonsets
```

### Using the Node Maintenance Operator

The __Node Maintenance operator__ enables cluster administrators to declaratively place nodes in maintenance by creating a `NodeMaintenance` custom resource.

```bash
cat node-maintenance-operator.yaml
```
```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: openshift-workload-availability
---
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  name: node-maintenance-operator
  namespace: openshift-workload-availability
---
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: node-maintenance-operator
  namespace: openshift-workload-availability 1
spec:
  channel: stable
  installPlanApproval: Automatic
  name: node-maintenance-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
  package: node-maintenance-operator
```


Apply the above YAML to your cluster and verify if the operator is installed and pod is running.

```bash
oc create -f node-maintenance-operator.yaml
```
```bash
oc get pods -n openshift-workload-availability
```

After the operator is successfully installed, it exposes a node maintenance API of kind `NodeMaintenance`.

<pre>
<b>oc api-resources | grep nodemaintenance</b>
nodemaintenances   nm   nodemaintenance.medik8s.io/v1beta1  false  NodeMaintenance
</pre>


Use the `NodeMaintenance` CRD to put a node into maintenance.

```yaml
apiVersion: nodemaintenance.medik8s.io/v1beta1
kind: NodeMaintenance
metadata:
  name: maintenance-in-progress  
spec:
  nodeName: worker02 
  reason: "Node Maintenance"
```
```bash
oc create -f node-maintenance.yaml
```


check the status by running the `oc get nm` command.

### Removing a Node from Maintenance Mode

To remove the node from maintenance mode and prepare it to accept new workloads, delete the `NodeMaintenance` resource.

```bash
oc delete NodeMaintenance maintenance-in-progress
```

### Eviction Strategies

Eviction strategies determine whether OpenShift moves the VMs on the failed or drained node to another node or terminates them. Three strategies can be configured.

- `LiveMigrate`

- `LiveMigrateIfPossible`

- `None`

They can be configured using the Web Console or the `oc` command.

```bash
oc edit vm <VMName> -n namespace
```
```yaml
...
spec:
  template:
    spec:
      evictionStrategy: LiveMigrateIfPossible
...
```

When you create a VM, the eviction strategy is set to `LiveMigrate` by default.


### Pod Disruption Budget

A Pod Disruption Budget (PDB) limits the number of Pods of a replicated application that are down simultaneously from voluntary disruptions (like node upgrades or autoscaling)

In OpenShift Virtualization, a PDB resource is created and deleted with each VMI to prevent the `virt-launcher` pod from being deleted before the live migration process is completed.

To create a PDB, create a file named `pdb.yaml`. You must configure __either__ `minAvailable` (minimum pods that must remain online) or `maxUnavailable` (maximum pods allowed to go offline). Do not use both in the same PDB.


```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-vm-pdb
  namespace: default
spec:
  minAvailable: 1
  selector:
    matchLabels:
      app: web-vm
```

> In OpenShift Virtualization, the `maxUnavailable` doesn't make sense as there is only one VM.

```bash
oc apply -f pdb-file.yaml
```

```bash
oc get pdb
```
