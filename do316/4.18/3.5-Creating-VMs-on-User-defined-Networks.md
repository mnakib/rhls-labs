### User-defined Networks

OpenShift Virtualization User-Defined Networks (UDN) are customizable network segments built on OVN-Kubernetes that allow you to create isolated, project-specific networks for Virtual Machines (VMs) and pods. Unlike the default shared pod network, a UDN enables you to define your own IP subnets and topology, providing a networking experience similar to traditional virtualization platforms like OpenStack or VMware.

In this scenario, we'll show two capabilities of user-defined networks:
- IP address persistence for VMs after reboots
- Two namespaces with identical subnets, with no conflict issues

#### Create the namespace

OpenShift requires this specific label (`k8s.ovn.org/primary-user-defined-network`) on any namespace that will use a UDN as its primary network. This label must be present at the time of namespace creation. 

```yaml
apiVersion: v1
kind: Namespace
metadata:
  name: vms-demo
  labels:
    # This label is MANDATORY for Primary UDN support
    k8s.ovn.org/primary-user-defined-network: ""
```

#### Define the UDNs

```yaml
apiVersion: k8s.ovn.org/v1
kind: UserDefinedNetwork
metadata:
  name: isolated-network
  namespace: vms-demo
spec:
  topology: Layer2
  layer2:
    role: Primary
    ipam:
      lifecycle: Persistent 
    subnets:
    - "10.70.0.0/24"
```

#### Create a VM inside the UDN namespace

Create a VM and see how its IP address doensn't change when restarting or migrating it

```bash
# Create a cloud-config file for storing the VM credentials
cat > cloud-config.yaml
```

```yaml
#cloud-config
user: admin
password: redhat
chpasswd: { expire: False }
ssh_pwauth: True
```

```bash
# Encode the cloud-config file
base64 -w0 cloud-config.yaml > cloud-config.b64
```

```bash
# Create a vm using the vms-demo namespace
virtctl create vm \
  --name=web \
  --instancetype=u1.small \
  --preference=rhel.10 \
  --volume-datasource=src:openshift-virtualization-os-images/rhel10-helloworld \
  --namespace vms-demo \
  --cloud-init-user-data "$(cat cloud-config.b64)"
```


There is an important concept on how to connect the VM to the UDN, which is 

<pre>
spec:
  <b>topology: Layer2</b>
  layer2:
    <b>role: Primary</b>
</pre>

The primary difference lies in how traffic reaches the physical world (Topologies) and how the workload treats the interface (Roles).

| Feature | topology: Layer2 | topology: Localnet |
|---|---|---|
| Network Type | Overlay (Virtual) | Underlay (Physical) |
| Connectivity | Creates a cluster-wide virtual switch using Geneve encapsulation. | Directly wires the virtual network to a physical network interface on the node. |
| Infrastructure | No physical network changes needed; works "out of the box" across nodes. | Requires OVS bridge mappings to be configured on every node. |
| Best For | Isolated internal traffic and seamless VM Live Migration. | Exposing workloads to external physical networks or specific VLANs. |
| Availability | Supported in UserDefinedNetwork (UDN). | Unavailable in standard UDN; only supported via ClusterUserDefinedNetwork or NetworkAttachmentDefinition. |

__Roles: Primary vs. Secondary__

| Feature | role: Primary | role: Secondary |
|---|---|---|
| Default Traffic | Acts as the default gateway (eth0) for the pod or VM. | Acts as an additional interface (e.g., net1) for specific traffic only. |
| Assignment | Automatically assigned to every workload in the namespace. | Manually assigned via annotations (e.g., k8s.v1.cni.cncf.io/networks). |
| Isolation | High. Workloads typically cannot talk to the default cluster network. | Multi-homing. Allows a workload to be on the default network and this UDN simultaneously. |
| Limit | Only one primary UDN is allowed per namespace. | A workload can have multiple secondary networks. |

In short: Use __Layer2__ with __Primary__ if you want a simple, isolated virtual network that replaces the default pod network for your VMs. Use __Localnet__ with __Secondary__ if you need your VMs to "sit" directly on an existing physical VLAN alongside their standard network.
