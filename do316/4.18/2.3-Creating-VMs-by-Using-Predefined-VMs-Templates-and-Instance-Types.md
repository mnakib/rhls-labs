

Creating VMs can be done by using predefined templates, instance types or YAML resource files.

### Virtual Machine Templates

A template is an OpenShift API resource that helps creating several resource types, including network interfaces, PVs, and VMs. Templates contain default parameters that you can customize according to your needs.

```bash
# Display all OpenShift templates, including apps and VMs
oc get templates -n openshift

# Display all rhel templates
oc get templates -n openshift | grep rhel

# Display the rhel9-server-small template
oc get templates rhel9-server-small -n openshift

# Describe the rhel9-server-small template
oc describe templates rhel9-server-small -n openshift | less
```

The template is creating a YAML file to create those described resources. To check the YAML used.

```bash
oc get templates rhel9-server-small -n openshift -o yaml | less

# Output the template content to the template_rhel9-server-small.yaml
oc get templates rhel9-server-small -n openshift -o yaml > template_rhel9-server-small.yaml
```

The resources listed below the `objects` part are the resources that will be create by this template. In this example, we have only one resource object, which a `VirtualMachine`.

<pre>
...
<b>objects:</b>
- apiVersion: kubevirt.io/v1
  <b>kind: VirtualMachine</b>
  metadata:
    annotations:
      vm.kubevirt.io/validations: |
...
</pre>

You can see the use of variables (aka. parameters) like `${NAME}`, `${NAMESPACE}`, `${DATA_SOURCE_NAMESPACE}` in different locations.

The are all listed at the end of the template file.

```yaml
parameters:
- description: VM name
  from: rhel9-[a-z0-9]{16}
  generate: expression
  name: NAME
- description: Name of the DataSource to clone
  name: DATA_SOURCE_NAME
  value: rhel9
- description: Namespace of the DataSource
  name: DATA_SOURCE_NAMESPACE
  value: openshift-virtualization-os-images
- description: Randomized password for the cloud-init user cloud-user
  from: '[a-z0-9]{4}-[a-z0-9]{4}-[a-z0-9]{4}'
  generate: expression
  name: CLOUD_USER_PASSWORD
```




### Virtual Machine Instance Types


Instance Types work with closely with them VirtualMachineClusterPreference

```bash
# List out all instance types
oc get virtualmachineclusterinstancetypes

# Describe the u1.small instance type
oc describe virtualmachineclusterinstancetypes u1.small
```
The specifications of this instance type if 1 cpu and 2G of memory.

<pre>
API Version:  instancetype.kubevirt.io/v1beta1
Kind:         VirtualMachineClusterInstancetype
Metadata:
  ...
<b>Spec:
  Cpu:
    Guest:  1
  Memory:
    Guest:  2Gi</b>
</pre>

When creating a VM using an instance type, you cannot go and change the cpu and memory allocation of the virtual machine. You have to update the instance type that it uses.

Now, let's have at the `VirtualMachineClusterPreference` resources, as Instance Types work with closely with them.

```bash
oc get virtualmachineclusterpreferences
```

A `VirtualMachineClusterPreference` describes should a VM behaves by default. It will have things related to which disks are attached to the VM, which features are enabled, and so on. 

Let's have closer look at those VM specifications by describing the `rhel.9` VM prefence.

```bash
oc describe virtualmachineclusterpreferences rhel.9 
```

Those specifications (or preferences) work in conjunction with data sources. A data sources specifies where the image comes from. 

```bash
oc get das -n openshift-virtualization-os-images
```

```bash
oc describe datasources rhel9 -n openshift-virtualization-os-images
```

If you see closely, you can see that the __rhel9__ `datasource` (or boot volume) is configured to work by default with the __u1.medium__ `VirtualMachineClusterInstancetype` and the __rhel.9__ `VirtualMachineClusterPreference`.

### Creating Virtual Machines


Create a VM named _instance-type-vm_ from an __rhel9__ instance type and data source.


Create a VM named _template-vm_ from the __rhel9-server-small__ template.


Create a VM from a VM yaml resource file.


```bash
cat virtualmachine-example.yaml
```

Create the VM and check its successfully created, then start it

```bash
# Check the VM is created
oc get vm

# Start it
virtctl start example-vm
```

Connect to the VM with user configured in its resource file

```bash
virtctl ssh rhel@example-vm -i ~/.ssh/id_rsa
```
